# P0.3 两阶段切换（Key verified → Cutover，无黑洞）

> 定案：A 为硬门槛（必须通过）；B 为加分项（尽力而为，不阻断）。

## 目标
当用户提交 OpenAI Key 并验证通过后，把系统从“provision responder 在前台兜底/引导”切换到“OpenClaw Gateway 主导对话”，并保证：
- 不出现黑洞（用户发 `hi` 不会无响应）
- 任一步失败可自动回滚
- 过程可观测、可审计

## 术语
- **Phase A（硬门槛）**：Key 通过 + WhatsApp 发送链路可用（send-test 成功）
- **Phase B**：停止 provisioning responder（或进入 delivered mode）
- **Rollback**：Phase B 后如果发现不可用，则立刻恢复 provisioning responder

## 状态机（建议）

### S0 WAITING_KEY
进入条件：用户已绑定 WhatsApp，但未提供 OpenAI Key。
动作：提示用户提交 key。

### S1 KEY_VERIFYING
进入条件：收到 key。
动作：用最小 OpenAI 调用校验 key（建议 `GET /v1/models`，10s 超时）。
失败 → 回到 S0，并提示“key 无效/额度/网络”。

### S2 WA_STRONG_CHECK (Phase A - part 1)
进入条件：key 校验通过。
动作（强检查）：
- `sudo -u ubuntu openclaw gateway status` 必须 OK
- WhatsApp connected/ready（由 status 输出判定）
失败 → 回到 S0/S1（视情况）并提示重试。

### S3 WA_SEND_TEST (Phase A - hard gate)
动作：向 **self-chat** 发送一条测试消息（必须拿到“发送成功”回执/调用成功）。
- 为什么：覆盖最核心的“黑洞”风险（看似 connected 但实际发不出去）。
失败 → 不进入 Phase B；保持 provisioning responder；提示用户稍后重试。

### S4 CUTOVER (Phase B)
动作：停止 provisioning responder（systemd stop）。

### S5 POST_CHECK
动作：再次跑 `openclaw gateway status` 强检查。
失败 → **Rollback**（立刻 start provisioning responder），并提示用户“已回滚”。

### S6 DELIVERED
动作：标记交付完成态；后续策略由 P0.4 约束（只 self-chat，外部联系人忽略）。

## B 加分项：inbound 回流确认（非阻断）
切换完成后，给用户提示：
- “请回复任意内容（例如 OK）确认已切换成功。”

在 N 秒（例如 60s）内收到 inbound：标记 Fully verified。
超时：保持 Verified-by-send，不回滚。

## 可观测性/审计（最低要求）
建议为每次 cutover 记录：
- t_key_received
- t_key_verified
- t_wa_status_ok
- t_send_test_ok
- t_provision_stopped
- t_post_check_ok
- outcome（success/rollback + reason）

并落盘到 events（DB）或 JSONL（文件），方便统计 P50/P95。

## 最小实现脚本
- artifacts: `p-site/artifacts/v0.1.6/scripts/cutover.sh`
- 硬门槛 A：send-test 使用 `openclaw message send --channel whatsapp --target SELF_E164`。

## 回滚原则（硬规则）
- **只要 Phase B 之后出现任何不确定性（status 失败、send 失败、超时） → 立即回滚**。
- 回滚后必须给用户明确提示（不要沉默）。
