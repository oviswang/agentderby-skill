## WhatsApp provisioning responder (Baileys layer) - implemented on sample machine
- Goal: user machines respond without LLM (no API key needed) during onboarding; later allow pass-through after delivery.
- Key fixes during debugging:
  - WhatsApp LID conversations: inbound `remoteJid` can be `@lid` (e.g. `171...@lid`) not `@s.whatsapp.net`. Must not filter these out.
  - Self-chat detection must support both `selfJid` and `selfLid` (normalize `selfLidRaw` like `171...:46@lid` to `171...@lid`).
  - Avoid reply loops: store sent message ids in-memory (cap set) and ignore echoed messages.
- Persistent state file (per uuid): `/opt/bothook/provision/data/wa/<uuid>/autoresponder_state.json`
  - `welcomeSent`, `welcomeSentAt`
  - `lastSelfNudgeAt` (60s throttle)
  - `promoSentTo{ remoteJid: ts }` (external promo only once)
  - caches `payUrl`, `expiresAt`, `paid`, `deliveryStatus`
- Copy/templates: `/opt/bothook/provision/copy.json` (config-driven; avoids editing JS for copy)
  - Default EN; will add zh-CN (zh-TW reserved).
