## G1. WhatsApp 扫码已成功，但 P 页未提示“绑定成功”
- 症状：手机端扫码成功；用户机 openclaw status 显示 WhatsApp LINKED；但 P 页 Bound WhatsApp 显示 unknown，bound_at 为空。
- 影响：欢迎词/Key 引导不触发或延迟；状态机可能被 linking timeout release 抢先释放。
- 根因假设：
  1) 用户机（Baileys）link 成功后，没有可靠回调/上报到 control-plane 写入 deliveries.wa_jid + bound_at。
  2) P 页依赖 DB 字段轮询，但 DB 未更新。
- 建议修复位置：
  - provision 服务（bothook-provision）或 openclaw WhatsApp provider：link 成功后必须触发“回写控制面”API。
  - control-plane api-server：提供 idempotent endpoint（/api/wa/bound or similar）写 deliveries.wa_jid/bound_at，并记录 WA_LINKED_CONFIRMED 事件。

## G2. linking timeout release 抢先发生（明明已经 linked）
- 症状：DB events 记录 QR_GENERATED 后短时间内出现 DELIVERY_LINKING_TIMEOUT_RELEASE / RECYCLE_*，导致 deliveries.status 进入 LINKING_TIMEOUT。
- 影响：页面/流程被释放，后续引导与交付不再继续。
- 根因假设：timeout 判定仅基于 DB 字段（wa_jid/bound_at）而不是用户机实际会话状态。
- 建议修复：timeout 释放前做一次最后确认：探测用户机是否已 linked（provision/gateway），若已 linked 则写库推进，而不是释放。

## G3. 已付费用户实例被错误回池（状态漂移）
- 症状：实例（例如 83）被手工改为 ALLOCATED 后又回到 IN_POOL；events 出现 stale pin cleared / lifecycle sync。
- 影响：P0 风险：付费用户机可能被再次分配。
- 根因假设：
  - watchdog/ops worker 在未识别“subscription active”时执行了回池/清 pin。
- 已修复（2026-02-26）：
  - delivery_watchdog: active subscription 视为 paid，不再 unpin/回池。
  - api-server ops worker: 在回池前检查 subscriptions.status，active 则禁止回池。
- 后续建议：将“subscription active => never return to pool unless explicit reclaim”写成全局 invariant，并在所有写回路径加单元测试。

## G4. OpenClaw 半安装/插件缺失导致 18789 不监听
- 症状：openclaw-gateway.service crashloop；日志提示 openclaw binary not found / plugins.allow 指向不存在插件；18789 不监听。
- 影响：机器被误判 READY 或需要人工介入。
- 根因假设：npm global install 可能留下 partial state；bootstrap 大量 `|| true` 吞掉失败。
- 已修复（2026-02-26）：
  - bootstrap: hard-validate openclaw bin exists + openclaw --version 可运行，否则 exit。
  - bootstrap: hard-validate bothook plugins 在 extensions 落盘，否则 exit。
  - healthcheck: 将 18789 + RPC probe 设为 READY 硬门槛。
- 建议：减少关键路径 `|| true`；写 BOOTSTRAP_VERSION 落盘便于回溯。

## G5. openclaw.json gateway block 会被后续 config mutation 丢失
- 症状：openclaw-gateway-start.sh 一直等待 config valid；18789 不监听。
- 影响：需要人工补 gateway 块。
- 已修复（2026-02-26）：bootstrap 增加“确保 gateway block 存在”的幂等修复。

## G6. 交付机应存在 OPENAI_API_KEY env（目标要求未落地）
- 症状：用户机上查不到 OPENAI_API_KEY 环境变量；DB delivery_secrets 清空后，模型仍能回复（说明模型通路可能不依赖该 key）。
- 影响：用户理解混乱（以为 key 被恢复/被内置）；交付承诺与实现不一致。
- 根因假设：key 仅保存在 control-plane delivery_secrets 中，未在 cutover_delivered 下发到用户机 systemd EnvironmentFile。
- 建议修复：
  - cutover_delivered.sh：将 key 写入 /etc/bothook/openai.env（600）并在 openclaw-gateway.service 引入 EnvironmentFile；重启并验收。
  - 或明确产品策略：使用平台代理，不下发 env key，并统一文案。
